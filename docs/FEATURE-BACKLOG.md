# ENTRAINMENT — Feature Backlog

## Overview

ENTRAINMENT is a music-reactive Three.js rail shooter whose feature ambitions — thirty-plus creatures and hazards, on-beat player abilities, biome-shifting visuals, multiple game modes, and a persistent meta layer — vastly outstrip its current scaffolding. Today nearly every system is hardcoded: entities branch per-type across five sites in `entities.js`, the event map emits one flat spawn per beat, music is sampled ad-hoc with two drifting beat cursors, the ship has a single verb (hold-to-fire), colours live in one flat `COLORS` object, and persistence is a lone high-score key. This backlog front-loads the foundational systems that convert that hardcoded sprawl into data plus small reusable parts, so the long tail of content features collapses into config rather than code. Build the groundwork first; the feature dimensions then become cheap, low-risk additions layered on stable seams.

## Technical Groundwork (build first)

### 1. Interactive Input + Action/Gesture Layer
**What it is.** A refactor of the poll-only `Input` (a steer scalar plus a firing bool) into an action model: per-frame press/release/held edges for named actions (fire, dash, ability1..n, brake, boost, parry) over a remappable keymap, gamepad polling with deadzone, optional discrete lane snapping when `LANES` is enabled, and optional pointer/touch free-aim. `getSteer`/`isFiring` stay for backward compatibility. `Game` calls `input.update(bus)` once per frame so gesture timing can compare press timestamps against beat phase. Shootable section gates piggyback on the entity registry — a `switch` entity on destroy sets a branch flag the spawn DSL reads.

**Why it unlocks many features.** It is the connective tissue for the entire interactive-input-agency dimension and for letting abilities and on-beat mechanics actually be *triggered*: discrete lane snapping, mouse/touch aim, gamepad/analog (accessibility), brake/boost throttle feeding `gameSpeed`, on-beat parry/dash windows (with the Audio Bus and Ability systems), shootable branching corridors, polarity switch, two-finger steering, and remappable controls. Without press-edges and timestamps, none of the rhythm verbs are buildable.

### 2. Component-Based Entity Behaviour Registry
**What it is.** A data-driven replacement for the bespoke-per-type branching in `entities.js`. Add `src/game/registry.js` exporting `ENTITY_DEFS` keyed by subtype; each def declares a pool/shape tag, geometry/material keys (resolved against the existing shared resources), half-extents `hx/hy/hz`, `shootable`, `hp`, a debris spec, and an ordered list of behaviour components. Components are tiny pure functions over `(e, ctx)` — `scrollZ`, `trackPlayer`, `wobble`, `beamAttack` (lifted from `_updateDrone`), `splitOnDeath`, `oscillateX`, `sweepBeam`, `closeGap`, `armOnBeat`. `spawn()` looks up the def and acquires from the shape pool; `update()` becomes one loop calling each def's components; `destroy()` reads the def's debris. The three existing types (pillar, cube, drone) migrate as the first three defs to verify behaviour is unchanged.

**Why it unlocks many features.** Today every new creature/hazard means new code in five places (`_makeX` factory, `_poolFor` dispatch, the `spawn()` switch at lines 111–114, the `update()` if/else chain at 164–173, and `destroy()` at 246–255). ### 3. Audio-Feature Bus
**What it is.** `src/audio/bus.js` — an `AudioBus(map)` with `update(songTime)` called once per frame in `updatePlaying` before subsystems. It computes and caches master RMS/centroid, piano/synth RMS (via the existing `sampleCurve`), a derived `bands={low,mid,high}`, a single authoritative `onBeat` cross-flag, `beatPhase` in [0,1), `timeToNextBeat`, and `beatIndex`. The bus is passed into `entities.update` (`ctx.bands/onBeat/beatPhase`), `effects.update` (reads cached values instead of resampling), and input gesture timing. `config.js` gains a `BANDS`/`FEATURE` block; real frequency bands in `analyze.js` are an optional follow-on, with the bus deriving from existing curves if absent.

**Why it unlocks many features.** Music data is currently sampled unevenly: `effects.js` resamples per curve per frame, and `index.js` (244–247) runs its own beat-cursor walk separate from `effects.js`'s (58–62) — two cursors that can drift, and nothing exposes beat *phase* or frequency bands. A single cheap feed enables all audio-reactive mechanics (On-Beat Fire Bonus, On-Beat Dash/Parry windows, Off-Beat Combo Decay, band-gated lane hazards), beat-phase juice, the metronome ghost reticle, beat-reactive HUD, and consistent beat-driven enemy AI — and it kills the duplicate-cursor drift bug as a bonus.

### 4. Spawn-Pattern DSL (waves, formations, lanes)
**What it is.** `src/audio/patterns.js` exporting named pattern functions `pattern(ctx) -> event[]`, layered over the existing event list. A `LANES` model (e.g. five x-positions) lives in `config.js` as the single source of truth for lane snapping and lane-aligned spawns. `generateEvents()` keeps its beat walk but, instead of inline pushes, rolls a weighted pick from a section→pattern-weight table and expands the chosen pattern into events — each still the existing `{time,type,subtype,x,...}` schema, so `analyze.js` and `spawn()` need no change. Patterns include `gate(openLane)`, `wall_pair`, `lane_row`, `zigzag(phraseLen)`, `swarm(n)`, and `burst(onsetTime)` keyed off the onset curve.

**Why it unlocks many features.** The event map is currently flat single events emitted one-per-beat from a `sin()`-derived x. Almost every interesting hazard and mode needs *coordinated* spawns: Closing Walls (paired L+R), Slalom Pillars, Laser Grid Curtain, Synchronized Pillar Gates, Beat-Stepper Swarm, Onset-Driven Bursts, Boss-Wave Finale. The DSL unlocks formation hazards, audio-reactive bursts and beat-locked quantization, call-and-response volleys, section-gated mechanic shifts, and the boss finale — and Daily/Endless/Mirror modes become re-parameterizations of the same generator (different seed, flipped lanes, looped window).

# ENTRAINMENT — Closing: Roadmap & Quick Wins

> Closing section of the feature backlog. The catalog (creatures, hazards, terrains, VFX, audio-reactive
> mechanics, abilities, scoring/meta, modes, input, UI, juice, accessibility, narrative, bosses) collapses
> onto a small set of groundwork seams. Two constraints frame everything: **single-track audio** (no real
> stem isolation — anything stem-gated is reframed or stubbed) and **X-only ship movement** (no Y axis today
> — every vertical-implying hazard is tagged **[needs G9]**). Sequence below is: ship the self-contained
> juice now, build the substrate, then spend it on set-pieces. Item names match the catalog exactly.

### Phase B — Foundational Groundwork (the multipliers)

The load-bearing seams. Build in dependency order; each one converts a slice of the catalog from
code-spelunking edits into pure data plus a few shared functions.

1. **Resolve G9 — Movement-Axis Decision (BLOCKER).** Decide stay-2D-on-a-line (re-spec vertical hazards to threaten via the X/lane band) vs. add a small Y/hop axis. Gates Crusher Pistons, Pendulum Beam Sweeper, Falling Debris Shower, Closing Walls (Iris Squeeze), Sentinel Eye (Gaze Beam), Tempo Tunnel, Apex Speed-Lock.
2. **G3 — Audio-Feature Bus + Beat-Window Primitive.** Single authoritative `onBeat`/`beatPhase`/`bands`, plus the reusable `withinBeat(±W)` helper. Fixes the duplicate-beat-cursor drift bug and feeds nearly all Audio-Reactive and Interactive verbs.
3. **G6 — Effect-Event Vocabulary + Post ShaderPass Stack.** One typed effect dispatcher (`color_flash`, `chromatic`, `glitch/datamosh`, `speed_lines`, `fog_volume`, `biome_swap`, `fov_punch`, `grid_shockwave`, `hitstop`, `section_title`) plus the post stack. Almost every VFX, Juice, and many Narrative ideas funnel through it. *(Gate behind quality + photosensitivity flags from day one — biggest GPU cost.)*
4. **G1 — Component-Based Entity Behaviour Registry.** Data-driven `ENTITY_DEFS` + composable behaviour components (`scrollZ`, `trackPlayer`, `beamAttack`, `splitOnDeath`, `oscillateX`, `sweepBeam`, `armOnBeat`, `toggleVisible`, `spawnChild`, `recordReplayX`, `link`). Seed-migrate pillar/cube/drone for zero behaviour change.

6. **G4 — Input + Action / Gesture Layer.** Edge-tracked named actions, remappable keymap, gamepad/analog, lane-snap, pointer/touch aim; gesture windows read G3's `beatPhase`.
7. **G5 — Player Ability / Loadout System.** `ABILITY_DEFS` with cooldown/cost/resource/`onBeatOnly`; converges every "Overdrive" variant and acts through clean seams (`bullets.fire(fireMode)`, `timeScale`, `entities.destroy`).
8. **G7 — Run-Config Object + Persistence Layer + Settings Overlay.** `RunConfig {mode, seed, modifiers, loadout, sectionRange, mirror, difficulty, songUrl}` + versioned `store.js` + a settings DOM/feature-flags object. Unlocks the entire Game-Modes and Scoring/Meta dimensions and is the safety gate for flash/bloom-heavy content.
9. **G8 — Content Registry (Palettes / Materials / Biomes).** `PALETTES` (incl. colourblind-safe + photosensitivity-clamped) and section-keyed `BIOMES` with a lerping `BiomeDirector`.
